## B6-1: Q/W 切換模式後自動聚焦金額輸入框（含 A 快捷鍵）

### 驗證步驟
1. **A 快捷鍵測試**：在 idle 首頁（未選學生），確認處於閒置狀態
2. 按 `A` 鍵 → 觀察是否直接進入收入/支出模式（expense_direction 畫面）
3. **Q 快捷鍵測試**（B6-1）：回到 idle → 按 `Q` → 確認進入「訂便當」模式，且**金額輸入框自動獲得焦點**
4. **W 快捷鍵測試**：在 idle → 按 `W` → 進入繳費模式（需已有今日訂單，見 B6-15），金額輸入框自動獲得焦點
5. **鍵盤在金額框內測試**：在金額輸入框 focus 時，分別按 Q/W/E → 金額框應直接輸入數字，**不觸發模式切換**

### 預期行為
- idle + 按 A → 直接進收入/支出模式
- idle + 按 Q/W → 進入對應模式 + 金額框自動 focus
- 金額框 focus 時，Q/W/E 視為數字輸入，不切換模式
- A 鍵在 text input / search input 中無效

### 邊界條件
- 已選學生的狀態下按 A → **不**進入 expense（無條件限制，視實作為準）
- A 鍵在 input focus 時抑制（同 Q/W/E）

## B6-2: Flash Banner 顯示櫃台淨現金

### 驗證步驟
1. 確認目前無任何收支記錄（RecentStrip 為空或只有訂餐記錄）
2. 點「新增 收入/支出」→ 選「收入」→ 輸入金額 300 → 選「收入」原因 → 輸入備註 → Enter → 確認
3. 觀察 flash banner
4. 再新增一筆收入 200（流程同上）
5. 再新增一筆支出 100（選「支出」，選原因，Enter，確認）
6. 觀察 flash banner 的「餘額」欄位

### 預期行為
- 收支記錄存在時，flash banner「餘額」顯示櫃台淨現金（收入 300+200 = 500，扣支出 100 = 400）
- **不再顯示 0**（舊 bug）
- `__cashier__` 的收支記錄會影響淨現金計算

### 邊界條件
- 無收支記錄時 → flash banner「餘額」顯示 0 或不顯示
- 只有收入 → 餘額 = 收入總和
- 只有支出 → 餘額 = 負的支出總和

## B6-3: 輸入備註後 Enter 不被 ConfirmBanner 吃掉

### 驗證步驟
1. 點「新增 收入/支出」→ 選「支出」→ 選原因 → 進到「備註」輸入畫面
2. 輸入任意文字（例如「測試備註」）
3. **按 Enter**（馬上觀察，不要點擊其他地方）
4. 確認 flash banner 是否跳出，且**不被 ConfirmBanner 立即 dismiss**

### 預期行為
- Enter 後，expense_commit 成功 → flash banner 跳出顯示收支記錄
- **不被 ConfirmBanner 立即覆蓋**（舊 bug：Enter 被 window keydown listener 吃掉，ConfirmBanner 直接 dismiss）
- 若 ConfirmBanner 有顯示，則需手動按 Enter 或點擊確認才消失

### 邊界條件
- 備註為空（未填必填欄位）→ Enter 不送出
- 按 Escape → 取消，不送出

## B6-5: ExpensePanel 方向/原因選單支援鍵盤導航

### 驗證步驟
1. 點「新增 收入/支出」→ 選「支出」→ 進入「選擇方向」畫面（收入/支出）
2. **測試方向選擇鍵盤導航**：
   - 按 `←` → 選中左側按鈕（會有 outline ring）
   - 按 `→` → 選中右側按鈕
   - 按 `Enter` → 確認當前選中的選項
3. 進入「選擇原因」畫面（多個按鈕）
   - 按 `←` / `→` 在按鈕之間移動，選中者有 outline ring
   - 按 `Enter` → 確認選中的原因
4. 按 `Escape` → 取消，回到 idle

### 預期行為
- `←` / `→` 可在方向/原因按鈕之間切換，選中者有視覺 ring
- Enter 確認當前選項
- Escape 取消流程
- 操作期間滑鼠點擊仍然可用

### 邊界條件
- 只有一個選項時 ← / → 不應造成錯誤
- 鍵盤和滑鼠操作互不影響

## B6-9: 總交易筆數不包含未繳費訂單

### 驗證步驟
1. 查閱「每日報表」或「帳本」頁面中的「總交易筆數」
2. 新增 3 筆訂餐記錄（訂便當但**不繳費**）
3. 新增 1 筆收入、1 筆支出
4. 觀察總交易筆數

### 預期行為
- 總交易筆數**不包含**純訂餐但 paidAmount=0 的記錄
- 只計算有現金流的交易（收入、支出、已繳費訂單）
- 3 筆未繳費訂單 + 1 筆收入 + 1 筆支出 → 總交易筆數 = 2（或依實際有、現金的記錄數計算）

### 邊界條件
- 部分繳費的訂單（paidAmount > 0 但少於 mealPrice）→ 計入
- 已繳費訂單（paidAmount = mealPrice）→ 計入

## B6-14: 今日開帳金額 = 昨日關帳金額（countedCash）

### 驗證步驟
1. **建立昨日關帳基準**：
   - 設定日期為昨天（使用 TopBar ◀▶ 調整日期）
   - 進行一筆收支記錄（隨便一個金額）
   - 關帳，記錄 countedCash 金額（例如 4000）
2. **測試今日開帳**：
   - 將日期切回今天
   - 觀察「每日開帳金額」的初始值
3. **交叉驗證**：
   - 進到 AdminScreen / 設定頁面，確認開帳金額

### 預期行為
- 今日開帳金額的**預設值**自動填入昨日關帳時的 countedCash
- **不再**硬編碼預設為 4000
- 若昨日未關帳 → fallback 到預設值（4000 或其他合理值）

### 邊界條件
- 昨日無關帳記錄 → 顯示預設值 4000
- 跨月關帳（如 5/31 關帳，6/1 開帳）→ 正確取前一日 countedCash
- 手動修改開帳金額 → 覆蓋預設值

## B6-15: 已訂過便當的學生再次選擇時預設繳費模式

### 驗證步驟
1. 搜尋一個今天**尚未訂過便當**的學生（例如「王小美」）
2. 按 Enter → 觀察預設模式（應為**訂便當 order mode**）
3. 正常完成一筆訂餐（選擇品項、輸入價格、Enter 確認）
4. 從 RecentStrip 確認訂單已記錄
5. **再次搜尋同一個學生** → 按 Enter 選擇
6. 觀察預設模式

### 預期行為
- 第一次選擇學生 → mode = `order`（訂便當）
- **第二次選擇同一學生**（今日已訂過便當）→ mode = **`payment`**（繳費）
- UI 預設切到繳費區塊，預填金額或預選繳費按鈕

### 邊界條件
- 學生今天有繳費記錄但無訂餐記錄 → 仍為 order mode
- 學生從未出現過 → 為 order mode
- 跨日後再次選擇 → 視為新的一天，恢復 order mode
