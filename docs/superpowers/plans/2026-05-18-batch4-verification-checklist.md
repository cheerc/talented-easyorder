## F4-1: 儲值 flash 餘額雙重計算

### 驗證步驟
1. 啟動 App，確認學生列表有帳戶餘額為 0 的學生
2. 搜尋該學生 → 點擊選擇 → 切到「繳費」模式（按 W 鍵或點擊）
3. 輸入繳費金額 500 → 按 Enter 確認
4. 觀察畫面頂端或中間的 flash banner（確認橫幅）

### 預期行為
- flash banner 顯示 `餘額 $500`（非 $1,000）
- `detail` 顯示 `繳費: 收現 500`
- 右邊 RecentStrip 的最新一筆顯示 `+500`（綠色）

### 邊界條件
- 餘額為 -200 的學生（欠款），儲值 500 → flash 顯示餘額 $300
- 餘額為 500 的學生，繳費 0 → flash 顯示餘額 $500（未變動）

## F4-2: 收支完成後狀態卡住

### 驗證步驟
1. 在 idle 首頁點擊「新增 收入/支出」按鈕
2. 輸入金額 150 → 按 Enter
3. 選「支出」→ 選「付便當錢」
4. 觀察 flash banner 出現
5. 點擊 flash banner 關閉按鈕（或等待倒數結束）
6. 再次點擊「新增 收入/支出」按鈕

### 預期行為
- 支出 commit 後出現 flash banner，顯示 `支出: 付便當錢 −$150`
- 關閉 flash 後回到 idle 狀態
- 「新增 收入/支出」按鈕可再次點擊，無需重整網頁
- 重複上述步驟 3 次，每次都能正常完成

### 邊界條件
- 連續輸入 2 筆收入 → 2 次 flash 都能正常 dismiss
- 收入 → 支出 → 收入交替輸入 → 不卡住
- expense flash 點擊「復原」→ transaction 被刪除，不影響學生帳戶

## F4-3: 收入流程跳過 reason 選擇

### 驗證步驟
1. 點擊「新增 收入/支出」
2. 輸入金額 500 → 按 Enter
3. 在方向選擇畫面，點擊「收入」
4. 觀察接下來出現的畫面
5. 再次操作一次，但選「支出」

### 預期行為
- **選「收入」**：直接進入備註輸入畫面（跳過 reason 選擇），可輸入備註後確認
- **選「支出」**：進入 reason 選擇畫面（付便當錢 / 其他原因）
- 收入 commit 後，RecentStrip 顯示 `+500`（綠色，badge 顯示「收」）
- 支出 commit 後，RecentStrip 顯示 `−150`（紅色，badge 顯示「支」）

### 邊界條件
- 收入備註可留空 → 仍可正常 commit
- 支出「付便當錢」→ 直接 commit，不經過備註輸入
- 支出「其他原因」→ 進入備註輸入

## F4-5: 今日帳顯示現金流記錄

### 驗證步驟
1. 輸入幾筆收支（至少 2 筆收入 + 2 筆支出）
2. 切換到「今日帳」tab（F2）
3. 觀察頁面上方 ReportSummaryStats 統計區
4. 向下捲動觀察 LedgerGroupedTable 的 expense section

### 預期行為
- ReportSummaryStats 新增「櫃台收入」「櫃台支出」兩欄：
  - 櫃台收入：顯示收入總額（綠色）+ 筆數
  - 櫃台支出：顯示支出總額（紅色）+ 筆數
- LedgerGroupedTable 頂端「櫃台 收入/支出」區塊：
  - 收入 row：淡綠背景，type 欄顯示「收入」(綠色粗體)，金額欄 `+$xxx` (綠色)
  - 支出 row：淡紅背景，type 欄顯示「支出」(紅色粗體)，金額欄 `−$xxx` (紅色)
  - 每一筆顯示備註內容和刪除按鈕

### 邊界條件
- 完全沒有收支記錄 → 不顯示「櫃台 收入/支出」區塊
- 只有收入 → type 欄全部顯示「收入」，櫃台支出統計為 0 筆
- 只有支出 → type 欄全部顯示「支出」，櫃台收入統計為 0 筆
- 訂餐學生的交易正常顯示在下方的分組表格中，不與櫃台收支混淆

## F4-6: RecentStrip order/payment 用 afterBalance 判斷

### 驗證步驟
1. 找一個餘額為 0 的學生 → 訂餐 90（不繳費）
2. 找一個餘額為 500 的學生 → 訂餐 90（不繳費）
3. 找一個餘額為 50 的學生 → 訂餐 90（不繳費）
4. 找一個餘額為 -400 的學生 → 訂餐 90（不繳費）
5. 找一個餘額為 -90 的學生 → 繳費 50

### 預期行為
| 場景 | afterBalance | 顯示 | 顏色 |
|------|-------------|------|------|
| 餘額 0，訂餐 90 | -90 | 待繳費 90 | 紅 |
| 餘額 500，訂餐 90 | 410 | 已繳費 90 | 綠 |
| 餘額 50，訂餐 90 | -40 | 待繳費 40 | 紅 |
| 餘額 -400，訂餐 90 | -490 | 待繳費 490 | 紅 |
| 餘額 -90，繳費 50 | -40 | 待繳費 40 | 紅 |

### 邊界條件
- 全額繳清 → `afterBalance = 0` → `>= 0` → 顯示「已繳費」(綠)
- 繳費 500 後 afterBalance >= 0 → 顯示 `+$500` (綠)
- order 的 paidAmount 不直接用於判斷，完全以 afterBalance 為準

## F4-7: 開帳金額可重複儲存

### 驗證步驟
1. 切換到「設定」tab（F3）
2. 在「每日開帳金額」輸入 4000 → 點擊「儲存開帳金額」
3. 觀察按鈕旁出現「已儲存」提示（2 秒後消失）
4. 再次修改為 5000 → 點擊「儲存開帳金額」
5. 觀察是否成功儲存

### 預期行為
- 第一次儲存：呼叫 openCashSession 建立 session → 成功
- 第二次儲存：呼叫 updateOpeningCash 更新 → 成功，顯示「已儲存」
- 第三次儲存：同上，可無限次更新

### 邊界條件
- 日期已關帳（dateStatus === 'closed'）→ 開帳金額 input 和儲存按鈕 disabled
- 輸入負數 → 不儲存（前端檢查 >= 0）
- 輸入非數字 → 不儲存（Number.isFinite 檢查）
- 關帳後重開（reopen）→ 可再次修改開帳金額

## 整合測試（全部 F4-1 ~ F4-7）

### 完整流程
1. 啟動 App → Admin 設定開帳金額 4000
2. 修改開帳金額為 5000 → 確認可儲存 → 「已儲存」
3. 回到 POS → 搜尋學生「王小美」（餘額 0）→ 訂餐 90 → 確認 flash 顯示「餘額 −$90」
4. 對王小美儲值 200 → flash 顯示「餘額 $110」（非 $310）
5. 新增收入 300（贊助金）→ 直接進備註 → 輸入「贊助金」→ commit
6. 新增支出 150（買文具）→ 選「支出」→ 選「其他原因」→ 輸入「買文具」→ commit
7. 觀察 RecentStrip：
   - 王小美儲值：badge「繳」，金額 `+$200`，afterBalance 110 → 綠
   - 收入：badge「收」(綠)，金額 `+$300`
   - 支出：badge「支」(紅)，金額 `−$150`
8. 確認「新增 收入/支出」按鈕仍可使用
9. 切到「今日帳」→ 確認：
   - 櫃台收入：$300 (1 筆)
   - 櫃台支出：$150 (1 筆)
   - expense section 分列收入/支出，各有備註
10. 回 POS → 王小美的 order 顯示「已繳費 90」（因 afterBalance = 110 >= 0）
11. 選「葉大雄」（餘額 0）→ 訂餐 90 → 顯示「待繳費 90」(紅)
12. 再對葉大雄繳費 50 → 顯示「待繳費 40」(紅)（因 afterBalance = -40）
13. 全部測試完成，無需重整網頁
